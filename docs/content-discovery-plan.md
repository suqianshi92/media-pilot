# 内容发现第一版计划

## 目标

在现有资源搜索页右侧增加一个轻量内容发现会话。用户可以用自然语言描述题材、风格、剧情或相似作品，系统用现有 LLM 配置流式返回 Markdown 片单建议和可复制的资源查询关键词。用户再把关键词复制到资源搜索框，走现有资源搜索、预检索元数据、下载和入库链路。

## 不做

- 不生成可点击片单卡片。
- 不在内容发现阶段解析 TMDB / TPDB 元数据。
- 不把内容发现会话写入数据库。
- 不复用入库任务 AgentRun / AgentMessage / AgentToolCall / AgentDecisionRequest。
- 不让搜索框走 LLM 意图解析或关键词优化。
- 不提供下载站点、磁力链接、规避限制方法或未授权内容获取步骤。
- 不为内容发现单独增加模型配置。
- 不读取本地已下载或已入库历史来避免重复推荐。

## 页面形态

- 页面：继续使用现有“资源搜索”页。
- 主区域：直接资源搜索。搜索框输入什么就查什么，不调用 LLM。
- 右侧：内容发现会话面板，复用任务 Agent 面板的视觉样式、Markdown 渲染、滚动体验和流式文本体验。
- 会话支持多轮上下文；用户可通过刷新 / 新会话按钮清空当前上下文。
- 桌面优先支持双栏布局，右侧宽度可拖拽调整。
- 移动端只保证不严重破版，不做专门体验设计。

## 内容发现输出格式

使用 Markdown 有序列表。固定字段，不固定条目数量；候选少时可以少给，不能为凑数编造。

```markdown
1. **片名**（年份）
   - 推荐理由：一句话说明为什么匹配用户偏好。
   - 推荐搜索词：关键词1、关键词2
```

提示词边界保持简短：

```text
你是 Media Pilot 的内容发现助手。只推荐可供用户自行选择的影视作品，并给出简短理由和资源查询关键词。不要提供下载站点、磁力链接、规避限制的方法或获取未授权内容的操作步骤。
```

内容发现只在用户意图和所用模型能力范围内提供文本推荐，不承诺覆盖所有媒体类型，也不绑定具体元数据源或元数据档案。

## API 形态

新增轻量接口，不接入任务 Agent runner：

```text
POST /api/v1/content-discovery/stream
```

请求体：

```json
{
  "messages": [
    {"role": "user", "content": "推荐几部现代西部片"}
  ]
}
```

响应：

- SSE 流式文本 delta。
- 前端维护临时消息历史。
- 第一版不做服务端持久化。

后端复用现有 OpenAI-compatible LLM 配置和客户端能力，但不创建任务、AgentRun、工具调用或人工决策。

## 资源搜索调整

- 现有资源页搜索框改为直接资源搜索。
- 移除或下线默认 LLM 意图解析模式。
- 不保留“优化关键词”入口。
- 用户不确定想看什么时，使用右侧内容发现会话。
- 用户复制内容发现给出的推荐搜索词后，粘贴到搜索框自行搜索。

资源结果上的现有“识别影片 / 预检索元数据”入口保留。预检索默认使用本次资源搜索关键词，用户仍可手动调整元数据候选。

## 验证

- 资源搜索框不调用 LLM，搜索响应只依赖 Prowlarr / indexer。
- 内容发现面板可流式输出 Markdown。
- 内容发现支持多轮上下文。
- 新会话按钮能清空当前上下文。
- Markdown 列表、加粗片名和推荐搜索词在右侧栏中可读且可复制。
- 资源搜索、预检索元数据、下载提交流程不因内容发现面板存在而回归。
- 窄屏下页面不出现严重重叠或不可滚动。

## 后续方向

如果纯文本内容发现确认有价值，再考虑第二版：

- 推荐完成后异步解析元数据源。
- 将文本建议升级为可点击内容发现条目卡片。
- 已解析条目携带元数据预选进入资源搜索。
- 卡片搜索使用多个资源查询关键词按顺序回退。
